## 1. プロジェクトの背景と目的

### 1.1 背景

AI時代の到来により、ブログ記事が検索結果の要約やAI生成コンテンツに埋もれ、「読まれない記事」が増えていくことが予想される。

Sayo's Journal は、ライター・本岡紗代（FUNE）が飯田市・下伊那郡エリアの「人・場所・文化・食」を丁寧に取材し、言葉と写真で届けるナラティブブログである。一つひとつの記事には、現地を訪ねた人にしか書けない温度感がある。

しかし、どれほど良い記事があっても、ユーザーが見つけられなければ読まれない。

### 1.2 目的

AIを「記事を奪う敵」ではなく「記事への入口をつくる味方」にする。

対話型AIを導入し、ユーザーが自然言語で質問するだけで、最適な記事にたどり着ける体験を提供する。

AIは記事の代わりではない。AIは「この記事を読んでみたい」と思わせるナビゲーターである。

### 1.3 コンセプト

「地域特化ブログ × 対話型AI」

- ユーザーはページ遷移なしで情報にアクセスできる
- ブログのデータベースをAIの知識源として活用する
- 詳しく知りたいユーザーは元記事に誘導される
- FUNEの人格を持つAIが、親しみやすく情報を届ける

### 1.4 プロジェクト体制

| 役割 | 担当 |
|------|------|
| エンジニア | 開発担当者 |
| ライター / コンテンツ管理 | 本岡紗代（FUNE） |

## 4. AIチャットページ

### 4.1 概要

| 項目 | 値 |
|------|-----|
| パス | `/` |
| レンダリング | Dynamic（キャッシュなし） |
| 主要機能 | 対話型AIチャット、タグクリック、記事カード表示 |

### 4.2 AIの人格

| 項目 | 内容 |
|------|------|
| 名前 | FUNE（紗代さんの分身） |
| トーン | 親しみやすく、体験ベースで語る |
| 一人称 | 私 |

特徴:

- 取材経験に基づいた語り口
- 温かみのある表現
- 押しつけがましくない提案

制約:

- 飯田市・下伊那郡エリア以外の質問には対応範囲外であることを伝える
- ナレッジデータにない情報は推測で答えず、正直に伝える

|------|------|

- ストリーミング表示（1文字ずつ流れるように表示）
- テキスト部分はストリーミングで表示

### 4.7 タグ機能

| 項目 | 内容 |
|------|------|
| 生成方法 | 事前生成（AIが50記事を分析して一括生成） |

タグの種類:

| 種別 | 例 |

### 4.8 レスポンシブ対応

| デバイス | レイアウト |

## 5. 会話機能の仕様

### 5.1 マルチターン（会話継続）

- 1セッション内で会話を継続できる
- 過去のやりとりをコンテキストとしてAIに渡す
- コンテキストウィンドウの上限を考慮し、古い会話は要約または切り捨て

### 5.2 会話履歴の保存

| 項目 | 内容 |
|------|------|
| 保存先 | sessionStorage（ブラウザのタブ単位） |
| ライフサイクル | タブを閉じたらリセット |
| サーバー保存 | なし |
| 将来拡張 | アカウント機能導入時にDB保存を検討 |

### 5.3 対応範囲外の質問への応答

飯田市・下伊那郡エリア以外の質問、またはナレッジデータに情報がない質問に対しては、FUNEの人格を維持したまま丁寧に断り、対応範囲を案内する。

応答例: 「ごめんなさい、松本市の情報はまだ記事にできていないんです。このサイトでは飯田市・下伊那郡エリアの情報をお届けしているので、そのあたりのことならお気軽に聞いてくださいね。」

### 6.7 RPC 関数

**match_articles(query_embedding vector, match_threshold float, match_count int) → TABLE**

- コサイン類似度でベクトル検索
- is_active = true の記事のみ対象
- 類似度スコア付きで返却

**check_usage_limit(p_session_id text) → jsonb**

- 日次ユーザー制限と月次サイト制限をチェック
- allowed（真偽値）、daily_remaining（残回数）、reason（制限理由）を返却

**log_ai_usage(p_session_id text, p_query text, p_token_input int, p_token_output int, p_matched_articles jsonb) → void**

- 使用ログを記録
- 日次/月次カウンターをインクリメント

## 7. API エンドポイント（新規）

### 7.1 POST /api/ai/chat

AIチャットのメインエンドポイント。ストリーミングレスポンス。

リクエストボディ:

| フィールド | 型 | 説明 |
|-----------|-----|------|
| message | string | ユーザーの質問テキスト |
| session_id | string | ブラウザ生成の匿名UUID |
| history | array | 過去の会話履歴（role: user/assistant、content: テキスト） |

レスポンス形式: text/event-stream（Server-Sent Events）

ストリーミングイベントの種類:

| type | content | 説明 |
|------|---------|------|
| text | string | AIの回答テキスト（逐次送信） |
| articles | array | 関連記事の配列（slug, title, thumbnail_url, excerpt） |
| spots | array | スポット情報の配列（name, address, phone） |
| suggestions | array | 関連タグのラベル配列 |
| done | 空文字 | ストリーミング完了 |

処理フロー:

1. 利用制限チェック（check_usage_limit）
2. ユーザーの質問を Embedding 化（text-embedding-004）
3. ベクトル検索で関連記事を取得（2〜5件）
4. システムプロンプト + ナレッジ + 会話履歴 + 質問 を Gemini に送信
5. ストリーミングでレスポンスを返却
6. 完了後、使用ログを記録（log_ai_usage）

レスポンス:

## 9. ナレッジデータの運用フロー

### 9.1 初回セットアップ

1. ライターが対象50記事を選定
2. 管理画面の「一括生成」で AIが記事HTMLからナレッジデータを生成
3. ライター + エンジニアがレビュー・修正
4. Embedding を一括生成
5. AIプロンプトタグを AI で一括生成 → レビュー・調整

### 9.2 記事更新時の運用フロー

1. ライターが記事を編集・保存
2. 管理画面のナレッジ一覧に「要更新」バッジが表示される
3. 管理画面でナレッジデータを開く
4. 右カラムで元記事との差分を確認
5. ナレッジデータを手動で更新・保存
6. Embedding が自動再生成される

### 9.3 記事追加時の運用フロー

1. ライターが新規記事を公開
2. 管理画面でナレッジデータを新規作成（AIによる下書き生成可）
3. レビュー・修正・保存
4. Embedding が自動生成される
5. 必要に応じてプロンプトタグを追加・再生成

## 10. 検索の仕組み

### 10.1 ベクトル検索（MVP）

| 項目 | 内容 |
|------|------|
| Embedding モデル | Google text-embedding-004（768次元） |
| 検索方法 | コサイン類似度（pgvector） |
| 検索対象 | article_knowledge.embedding（is_active = true のみ） |
| 取得件数 | 上位 2〜5件（類似度スコアの閾値で動的に調整） |

検索フロー:

1. ユーザーの質問を text-embedding-004 で Embedding 化
2. match_articles RPC でベクトル検索
3. ヒットした記事のナレッジデータ（metadata + content）をAIに渡す

### 10.2 将来拡張: ハイブリッド検索

- キーワード検索（既存の tsvector + trigram）とベクトル検索を併用
- RRF（Reciprocal Rank Fusion）でスコアを統合
- 記事数が100件を超えた段階で検討

## 11. 利用制限

### 11.1 ユーザー日次制限（C方式）

| 項目 | 内容 |
|------|------|
| 識別方法 | localStorage に保存した匿名 session_id |
| 制限 | 1ユーザー1日あたり N 回（初期値は実装時に設定、目安30回） |
| リセット | 毎日 0:00（JST） |

制限到達時の表示: 「今日はたくさんお話しましたね！また明日お気軽に聞いてください。ブログページからも記事を探せますよ。」＋ ブログページへのリンク

### 11.2 サイト月次上限（D方式 — セーフティネット）

| 項目 | 内容 |
|------|------|
| 制限 | 月間の総API呼び出し回数 or トークン数に上限を設定 |
| 上限値 | 管理画面から設定可能 |

上限到達時: AIチャットの入力欄を無効化し「ただいまメンテナンス中です。ブログページから記事をお探しください。」＋ ブログページへのリンクを表示。ブログページ、記事詳細、検索は通常通り動作。

### 11.3 API障害時

Gemini API がダウン or タイムアウト時: 「申し訳ありません、ただいま混み合っています。ブログページから直接記事をお探しください。」＋ ブログページへのリンク
