Landing Page Audit

「最大5投稿分のSNS情報を、1本のWeb記事に」LP
— 既存フォーム／既存LPとの矛盾点

新しく作ろうとしている index_SNS5web.html と、すでに動いている /request/post のLP・投稿フォームを突き合わせた結果です。

対象：index_SNS5web.html（271行） 比較：/request/post 2026-07-11

いちばん大きな問題

LPは「メモ・箇条書き・チラシでOK」と何度も約束していますが、いまのフォームはSNS投稿のURL（必須・1〜5件）しか受け取れません。自由記述欄は2026-07-07にDBのカラムごと削除済みです。このまま公開すると、読者はスクロールしてフォームに着いた瞬間に「メモを書く欄がない」ことに気づきます。

矛盾点（重い順）

  1. メモ・箇条書き・チラシを送る欄が存在しない

    要修正
    LPの記述

    「投稿のメモ、箇条書き、写真、チラシ、SNSリンクなどをもとに、読みやすく整理します」

    「文章がまとまっていなくても送れますか？ → はい。箇条書き、メモ、SNSリンク、チラシの内容などでも大丈夫です」

    実装の実際

    送信できるのは SNS投稿のURL（必須・1〜5件）／種別／「その他」補足200字／電話・メール／公開希望／画像／同意 のみ。自由記述欄はありません。

    src/lib/inquiry-schema.ts:47（sns_urls は .min(1) で必須）／ supabase/migrations/20260707120000_drop_mini_inquiries_message.sql（message カラムを DROP 済み）

  2. チラシは送れない（画像2枚まで・PDF不可）

    要修正
    LPの記述

    「写真、チラシ」を送れるものとして繰り返し提示。「最大5投稿分」の情報量を前提にしている。

    実装の実際

    画像は2枚まで。許可形式は jpeg / png / webp / heic / heif で、PDFは弾かれます。5投稿分を謳って写真2枚は量的にも合いません。

    src/lib/inquiry-schema.ts:132-140（INQUIRY_IMAGE_MAX_COUNT = 2）

  3. 「最大5投稿分」の意味がズレている

    要修正
    LPの記述

    「投稿1＝お店の特徴／投稿2＝営業日・時間／投稿3＝昼と夜の使い方／投稿4＝店主の人柄／投稿5＝基本情報」＝5テーマ分の情報を書いて送るように読める。

    実装の実際

    すでに投稿済みのSNS投稿のURLを最大5本貼るだけ。数字の5は偶然一致していますが、行為が別物です。「投稿1に営業日を書く」ことはできません。

  4. 「無料記事」という名称を使っている（命名ルール違反）

    要修正
    LPの記述

    「無料記事の使い方」（ヒーロー）／「無料記事は、地域の情報を集める入口です」／「要約・整理した無料記事を作成し」／CTAボタン「無料記事の情報を送る」— 計4箇所。

    決めてあるルール

    2026-07-08に「名称としての『無料記事』は全廃 → 『投稿記事』に統一」と決定済み。「無料」を使ってよいのは訴求として効く4箇所（ヒーロー主コピー／「なんで無料で掲載できるの？」見出し／同本文／送信欄の注記）だけ。

    CLAUDE.md ／ .claude/rules/implementation-status.md に明記

  5. 送れる本数の約束が食い違う

    要判断
    新LPの記述

    「1投稿分の小さなお知らせを1本の記事にする使い方でも大丈夫です」→ 小さいものを何度でも送っていいように読める。

    既存LPのFAQ

    「何本まで送れますか？ → 同一の個人・団体につき月1本を目安に受け付けます」

  6. 対象読者とトーンが別物（同じフォームなのに）

    要判断
    新LP

    「初めて見る人にも分かる紹介記事に整えます」「店主の人柄」「お店だけでなく団体・イベントにも」＝事業者・団体の集客ツール

    /request/post

    「あなたの言葉が、地域の記録になる」「告知も、自慢も、失敗談も」「地域の掲示板」＝地域の個人

    矛盾というより「同じフォームなのに別商品に見える」問題。2枚を並べるのか、1枚に統合するのかの判断に直結します。

  7. 掲載条件・対応エリアの文言が二重管理になる

    確認
    新LP

    掲載NG＝「地域との関係が薄いもの、公序良俗に反するもの、事実確認が難しいもの」。対応エリア「飯田市・下伊那地域」を明記。

    /request/post

    掲載NG＝「誹謗中傷、権利関係が不明な画像、事実確認ができない内容、地域との関係が薄い内容」。対応エリアのFAQ項目は無し。

    対応エリアの明示は新LPのほうが親切です。既存LPにも足すのがよさそうです。

  8. 実装上の衝突（作るときの落とし穴）

    確認

    HTMLのクラス名（header hero card flow faq banner cta-box…）は既存2LPと完全に同名。.lp-post / .lp-long と同じく3つ目のスコープ（例 .lp-sns5）で必ず包まないと既存LPが壊れます。

    :root に定義された色変数（--pink:#D86C9B 等）もLP間で衝突するため、スコープクラス側に定義し直す必要があります。

    CTAボタンが href="#" でフォーム未接続。サイト本体（トップ／ブログ／ギャラリー／About）への内部リンクがゼロでSEO的に孤立します（既存2LPは内部リンクあり）。

    Google Fontsを @import で外部読み込み（Kaisei Opti / Nunito）。既存LPは next/font でセルフホストしているので方針を揃える必要があります。

先に決めること

A. LPを実装に合わせる（文言を直す）

「もう投稿済みのSNS投稿を、最大5本ぶんURLで送るだけ」と言い切る。メモ・チラシの記述を削る。DB変更なし、いちばん早い。

B. フォームに自由記述欄を戻す

message カラムを nullable で復活させ、メモ・箇条書きを書けるようにする。2026-07-07 に「SNSに投稿している人に絞る」として消した判断を巻き戻すことになる。

C. 両方（URL必須のまま、任意の補足メモ欄を足す）

SNS URLは必須のまま、任意の「補足メモ」欄を追加。DBに1カラム追加。ハイブリッド。

そのうえで、ページ構成の判断
  • 独立LPとして新規作成 — フォームは既存の submitMiniInquiry を再利用。お店や団体に直接URLを渡す営業ツールとして使うなら、これが最適。
  • /request/post に事例セクションとして統合 — 「飲食店なら／朝活なら／市民活動なら」の3事例を既存LPに足す。検索やサイト内から来る人が主なら、1枚を強くするほうが効く（既存LPは理念が厚く「で、何を送ればいいの？」が薄い）。
  • /request/post からリンクする — 迷っていた案。LPからLPへ送るとフォームまでの距離が伸びるため、通常はコンバージョンが落ちる。おすすめしません。
判断の分かれ目は「誰に、どこで見せるか」です。直接渡す営業ツールなら独立LP、検索・サイト内導線なら統合。まずここを決めれば、あとは文言調整で片づきます。
